## Composable orchestration

NimbleBrain's core value is **composing multiple MCP servers into a single, orchestrated workspace**. Rather than connecting to one MCP server at a time, NimbleBrain aggregates tools from every installed app into a unified namespace, layers prompts to give the agent awareness of all available capabilities, and filters tool access per-task through skills.

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ                  ToolRegistry                     в”Ӯ
в”Ӯ                                                   в”Ӯ
в”Ӯ  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ в”Ӯ
в”Ӯ  в”Ӯ App 1   в”Ӯ  в”Ӯ App 2   в”Ӯ  в”Ӯ App 3 (remote)   в”Ӯ в”Ӯ
в”Ӯ  в”Ӯ (stdio) в”Ӯ  в”Ӯ (stdio) в”Ӯ  в”Ӯ (HTTP/SSE)       в”Ӯ в”Ӯ
в”Ӯ  в”Ӯ 3 tools в”Ӯ  в”Ӯ 5 tools в”Ӯ  в”Ӯ 2 tools          в”Ӯ в”Ӯ
в”Ӯ  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ в”Ӯ
в”Ӯ                                                   в”Ӯ
в”Ӯ  Unified namespace: 10 app tools + system tools   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
          в”Ӯ                        в”Ӯ
          в–ј                        в–ј
   Skill filtering          Agent Engine
   (allowed-tools)          (agentic loop)
```

Three mechanisms make this composable:

| Mechanism | What it does |
|-----------|-------------|
| **Tool aggregation** | The `ToolRegistry` collects tools from every MCP source (local stdio, remote HTTP/SSE) and presents them as a flat namespace. The agent doesn't know or care which server owns a tool. |
| **Skill-scoped filtering** | Each skill declares `allowed-tools` glob patterns. When a skill matches, only tools matching those patterns (plus system tools) are visible to the agent. This scopes the agent's capabilities per task. |
| **4-layer prompt composition** | System prompts are assembled from: (1) identity, (2) core context, (3) installed app metadata with trust scores, and (4) the matched skill's prompt. Every layer adds awareness without the agent needing to discover it. |

<Aside type="tip">
This composition is what distinguishes NimbleBrain from a single-server MCP client. You can install a Postgres app, a web search app, and a CRM app вҖ” then write a skill that gives the agent access to all three for a "lead enrichment" task, while a different skill restricts it to just Postgres for "database admin" tasks.
</Aside>

## Agent loop

Every chat message triggers an agentic loop in the `AgentEngine`. The loop repeats until the model produces a response with no tool calls, or a limit is reached.

```
User message
     в”Ӯ
     в–ј
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  Call Claude     в”Ӯв—„в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  (with tools)   в”Ӯ                   в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ                   в”Ӯ
         в”Ӯ                            в”Ӯ
    в”Ңв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”җ                       в”Ӯ
    в”Ӯ Tool    в”Ӯ  Yes                  в”Ӯ
    в”Ӯ calls?  в”Ӯв”Җв”Җв”Җв”Җв”Җв”Җв–ә Execute tools  в”Ӯ
    в”Ӯ         в”Ӯ        in parallel в”Җв”Җв”Җв”ҳ
    в””в”Җв”Җв”Җв”Җв”¬в”Җв”Җв”Җв”Җв”ҳ
         в”Ӯ No
         в–ј
   Return response
```

Each pass through the loop is one **iteration**. Defaults:

| Setting | Default | Hard cap |
|---------|---------|----------|
| Max iterations | 25 | 50 |
| Max input tokens | 500,000 | вҖ” |
| Max output tokens | 16,384 | вҖ” |
| `models.default` | `anthropic:claude-sonnet-4-6` | вҖ” |

The engine stops for one of three reasons:

- **`complete`** вҖ” the model responded without requesting any tool calls
- **`max_iterations`** вҖ” the iteration limit was reached
- **`token_budget`** вҖ” cumulative input tokens exceeded `maxInputTokens`. When this happens, tool calls from the current LLM response are dropped (not executed) to avoid running tools whose results can't be processed.

All tool calls within a single iteration run in parallel via `Promise.all()`.

### Orchestration flow

Before the engine loop starts, the `Runtime` orchestrates the composition described above:

1. Resolve or create a conversation
2. Match the user message to a skill (triggers, then keywords)
3. Compose the system prompt вҖ” 4-layer: identity + core skills + installed apps + matched skill
4. Filter the unified tool namespace based on the matched skill's `allowed-tools` globs
5. Run the `AgentEngine` loop against the composed tool set
6. Persist the conversation (JSONL or in-memory)

## Bundles (apps)

A **bundle** is an MCP server package installed into NimbleBrain. Bundles follow the [MCPB](https://github.com/modelcontextprotocol/mcpb) format вҖ” each has a `manifest.json` declaring how to start the server, what tools it provides, and optional UI metadata.

### Sources

Bundles are installed per workspace and tracked in `workspace.json` (not `nimblebrain.json` вҖ” bundle entries placed there are stripped on load). They come from:

| Source | How it works |
|--------|-------------|
| Registry | Downloaded from the [mpak](https://mpak.dev) registry by scoped name (e.g. `@nimblebraininc/ipinfo`). Managed across the org via `manage_apps`. |
| Connector catalog | Remote MCP connectors installed point-and-click from the in-app catalog, or via `manage_connectors`. |
| Local path | A bundle directory on the filesystem, for local development. |

### Lifecycle states

Each bundle process goes through these states:

```
starting в”Җв”Җв–ә running в”Җв”Җв–ә crashed в”Җв”Җв–ә dead
                в”Ӯ                     в–І
                в””в”Җв”Җв–ә stopped в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

| State | Meaning |
|-------|---------|
| `starting` | Process is spawning |
| `running` | Healthy and serving tools |
| `crashed` | Process exited unexpectedly (will attempt recovery) |
| `dead` | Repeated crashes, not restarting |
| `stopped` | Manually stopped via API or CLI |

### No bundles by default

NimbleBrain does not install any MCP bundles by default. Platform capabilities (home, conversations, files, settings, usage, automations) are built in as inline tool sources. Install bundles from the [mpak registry](https://mpak.dev), a local path, or a remote URL to give the agent domain-specific tools.

### Bundle options

```json
{
  "name": "@nimblebraininc/postgres",
  "env": { "DATABASE_URL": "postgres://localhost/mydb" }
}
```

| Field | Description |
|-------|-------------|
| `env` | Environment variables passed to the bundle process |
| `trustScore` | MTF trust score (0вҖ“100) from the mpak registry |
| `ui` | UI metadata: name, icon, and placement declarations |

## Skills

Skills are Markdown files with YAML frontmatter. They control what the agent knows and what tools it can use for a given message.

### Skill file format

```markdown title="research.md"
---
name: research
description: >
  Deep research on a topic вҖ” search the web, gather sources, and synthesize a
  report. Use when the user asks to research, investigate, or analyze a topic.
allowed-tools: websearch__search nb__search
metadata:
  nimblebrain:
    loading-strategy: dynamic
    priority: 50
    triggers:
      - "research this"
      - "deep dive"
---

You are a research agent. Search the web, gather multiple sources,
and synthesize findings into a structured report.
```

### How a skill loads вҖ” `loading-strategy`

| `loading-strategy` | Priority | Behavior |
|--------------------|----------|----------|
| `always` | 0вҖ“10 (core) or 11вҖ“99 | Always composed into the system prompt вҖ” identity, voice, durable rules. |
| `dynamic` | 11вҖ“99 | Loaded on demand, via the activation signals below. |

### How dynamic skills activate

A `dynamic` skill enters context through any of:

- **Description** вҖ” its name + description sit in a catalog; the model activates the relevant one (so the description must say *what it does and when to use it*).
- **Tool-affinity** вҖ” auto-activates when a tool in the active set matches one of its `tool-affinity` globs.
- **Triggers** вҖ” auto-activates on an exact phrase, for deterministic must-fire cases.

```
User: "research this topic and analyze the sources"
                    в”Ӯ
        trigger "research this" в”Җв”Җ match! в”Җв”Җв–ә research skill activates
```

An activated skill injects its markdown body into the system prompt; `allowed-tools` (if set) scopes which tools it may call.

### Skill directories

Skills are loaded from three locations (in order):

1. **Built-in** вҖ” `src/skills/builtin/` (shipped with the package)
2. **Global** вҖ” `~/.nimblebrain/skills/`
3. **Config** вҖ” directories listed in the `skillDirs` config option

## Tools

Tools come from two places: MCP bundles and built-in system tools.

### System tools

Every NimbleBrain instance has these `nb__*` tools available:

| Tool | Description |
|------|-------------|
| `nb__search` | Search installed tools (`scope: "tools"`) or the mpak registry (`scope: "registry"`) |
| `nb__status` | Platform status; `scope: "bundles"` for per-app health, `scope: "skills"` for loaded skills, `scope: "config"` for model and limits |
| `nb__delegate` | Spawn a child agent for sub-tasks (multi-agent delegation) |
| `nb__read_resource` | Read an MCP resource by URI |
| `manage_connectors` | List the connector catalog, install/disconnect remote MCP connectors per workspace or per user |
| `manage_apps` | Org-level management of installed registry apps: list, check for updates, upgrade |
| `manage_workspaces` | Create and manage workspaces |
| `manage_members` | Manage workspace membership |
| `manage_users` | Manage users (when a managed identity adapter is configured) |

App-platform capabilities are exposed through prefixed tool families: `conversations__*`, `files__*`, `automations__*`, `skills__*`, `usage__report`, `instructions__write_instructions`, and `compose__effective_context`.

### Tool surfacing strategy

NimbleBrain uses tiered tool surfacing to keep the LLM's tool list manageable:

| Condition | What the LLM sees |
|-----------|-------------------|
| Total tools вүӨ 30 | All tools surfaced directly |
| Total tools > 30, no skill matched | Only `nb__*` system tools. Others available via `nb__search` (`scope: "tools"`). |
| Skill matched with `allowed-tools` | Tools matching the globs + system tools. Others via `nb__search`. |

The threshold is configurable via `maxDirectTools` (default: 30).

<Aside type="tip">
When you have many bundles installed, the agent uses `nb__search` to find the right tool before calling it. You don't need to manage this вҖ” the bootstrap skill teaches the agent how to use meta-tools automatically.
</Aside>

### Multi-agent delegation

The `nb__delegate` tool spawns a child `AgentEngine` run with a scoped system prompt and filtered tool set. Named agent profiles are configured in [`workspace.json`](/config/workspace-json):

```json
{
  "agents": {
    "researcher": {
      "description": "Deep research agent",
      "systemPrompt": "You are a research agent...",
      "tools": ["websearch__*"],
      "maxIterations": 8,
      "model": "reasoning"
    }
  }
}
```

The child agent's iteration budget is capped at `min(child.maxIterations, parent.remaining - 1)`. Multiple delegations in the same turn run concurrently.

## Conversations

Conversations persist across messages so the agent remembers context.

### Storage

Conversations are workspace-owned. Every turn is written to a per-conversation JSONL file under the workspace it ran in:

```
~/.nimblebrain/workspaces/<wsId>/conversations/<ownerId>/<convId>.jsonl
```

The owner (`ownerId`) is the authorization principal вҖ” only they can read or resume the conversation вҖ” while the workspace (`wsId`) is where it is stored. Persistence does not depend on any top-level `store` setting; programmatic/SDK embedding can supply an in-memory `ConversationStore` instead.

### JSONL format

Each conversation is a single `.jsonl` file:

```
{"id":"conv_abc123","createdAt":"2025-03-15T10:30:00Z"}
{"role":"user","content":"What files are in my home directory?","ts":"..."}
{"role":"assistant","content":"Let me check...","ts":"...","toolCalls":[...]}
```

Line 1 is metadata (ID, creation timestamp). Lines 2+ are `StoredMessage` objects with role, content, timestamp, and optional tool call records.

### Resuming conversations

To continue an existing conversation, open it from the Conversations sidebar in the web UI. Through the API, pass `conversationId` in the chat request body.

## MCP Apps

NimbleBrain is a full [MCP Apps](https://modelcontextprotocol.io/extensions/apps/overview) host, implementing the [ext-apps specification](https://apps.extensions.modelcontextprotocol.io/api/). MCP Apps are interactive UI applications вҖ” built with HTML/JavaScript вҖ” that render directly inside the host as sandboxed iframes.

Unlike traditional web apps, MCP Apps:

- **Live inside the conversation** вҖ” no tab-switching, context stays together
- **Call tools bidirectionally** вҖ” apps invoke MCP tools and receive pushed results from the agent
- **Inherit the host's theme** вҖ” CSS variables are injected automatically
- **Run in a security sandbox** вҖ” no access to the parent page, cookies, or other apps

Any MCP bundle that declares UI resources and placements becomes an MCP App. See [MCP Apps](/concepts/mcp-apps) for the full concept and [MCP App Bridge](/apps/bridge) for the implementation protocol.

## Platform as MCP server

NimbleBrain exposes a `/mcp` endpoint that turns the platform into a Streamable HTTP MCP server. External MCP clients вҖ” Claude Code, Open WebUI, or another NimbleBrain instance вҖ” can connect through a single endpoint. Each request names its workspace via the `X-Workspace-Id` header (validated against your membership): `/mcp` serves that one workspace's tools plus your identity tools (conversations, files, automations), and refuses tools in any other workspace. A request with no `X-Workspace-Id` is identity-only.

This means NimbleBrain is both an MCP **client** (connecting to installed bundles) and an MCP **server** (exposing composed tools to external hosts). See [MCP Endpoint Reference](/connect/mcp-endpoint) for configuration and usage.
